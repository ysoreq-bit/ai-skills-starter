# Capability Promotion Policy

## Purpose

This policy defines how private AI-capability work can mature into this public starter without leaking private context or turning rough experiments into public guidance too early.

## Layer model

Use four distinct layers:

1. **Source layer:** authorized source systems, uploaded files, and original materials.
2. **Memory layer:** optional private study memory with distilled, source-linked academic records.
3. **Private capability incubator:** private workflows, prompts, routing experiments, evaluations, and personal operating rules.
4. **Public method layer:** this repository, containing only sanitized, stable, reusable methods.

The private capability incubator is the preferred place to develop and refine user-specific AI workflows. This public repository is a publication target, not the default workspace for every new idea.

## What belongs in a private capability incubator

Private capability work may include:

- rough workflow drafts;
- personal operating preferences;
- private evaluation notes;
- course-adapted method variants;
- examples from real work, when legally and privately appropriate;
- experiments that may fail;
- routing rules that depend on private repositories or tools;
- lessons that are useful to the user but not yet general.

Keep raw protected source material, secrets, and credentials out of the incubator unless a separate private policy explicitly allows and protects them.

## Promotion gate

Promote a capability into this public repository only when all checks pass:

1. The method is reusable beyond one private course, learner, account, machine, or repository.
2. Private course names, learner details, local paths, Drive identifiers, credentials, and private links are removed.
3. Real private examples are replaced with synthetic, public, or fully redacted examples.
4. The capability has a clear trigger and negative trigger.
5. The capability was tested on at least one original private case and one independent general case.
6. A negative or non-activation case was checked.
7. Overlap with existing public Skills, routes, and docs was reviewed.
8. Licensing and source provenance are acceptable for public release.
9. The promoted version is final or distilled, not a draft transcript.
10. The diff is small enough to review and can be reverted safely.

If any check fails, keep the capability private or promote only a smaller sanitized rule.

## Promotion workflow

1. Develop and revise privately.
2. Distill the transferable method.
3. Run a privacy and source-material review.
4. Replace private examples with safe fixtures or prose.
5. Add or update the smallest public Skill, route, registry entry, or documentation section.
6. Keep lifecycle status at `draft` or `candidate` until evaluation evidence supports promotion.
7. Record what was promoted and what intentionally stayed private.

## Prohibited promotion behavior

Do not:

- copy private study-memory records into this public repository;
- publish learner-model content, performance history, private source links, or course-specific notes;
- hard-code a private repository name, account, credential, or local path;
- promote a method solely because it was useful once;
- publish a full chat transcript when a concise method is enough;
- make this public repository depend on access to a private incubator.

## Acceptance checks

- A reader can use the public method without private repositories.
- The method still works when private memory and private capabilities are unavailable.
- Public examples contain no private source contents or identifiers.
- The route activates for the intended task and stays inactive for irrelevant tasks.
- The private incubator remains the place for ongoing experimentation.
